# eBird/GBIF 本地下载数据摸底

## 结论概览

- 现有数据包含一个 GBIF Simple CSV 下载：`0009321-260519110011954.csv`，大小约 15.08GB；同名 zip 约 3.91GB，压缩包内只有这一份 CSV。该数据不在当前目录下。当前目录仅包含小部分中国记录样例
- 文件扩展名是 `.csv`，但实际是 **UTF-8 + 制表符分隔 TSV**。读取时应使用 `delimiter="\t"`，不要按逗号 CSV 处理。
- 表头共有 50 个字段，形态是 GBIF/Darwin Core occurrence 记录；每行代表一次物种出现/观测记录。
- 基于前 200,000 行平均行宽估算，全文件约 27,654,025 条记录。这个数字是估算值；精确行数需要顺序扫完整个 15GB 文件。
- 数据记录类型 `basisOfRecord` 在样本中全为 `HUMAN_OBSERVATION`，符合 eBird 观测数据特征。

## 推荐使用方式

15GB TSV 不建议用 Excel 直接打开。建议按目标使用以下方式：

- 小规模验证：Python `csv.DictReader(..., delimiter="\t")` 流式读取前 N 行。
- 条件过滤：逐行扫描，只保留目标国家、物种、年份或经纬度范围，输出为较小 TSV/Parquet。
- 交互分析：先转换为 Parquet，再用 DuckDB/Polars/R data.table 查询。
- 地图分析：使用 `decimalLatitude`、`decimalLongitude`，但先检查 `issue` 中的坐标相关标记。
- 物种统计：优先用 `speciesKey`/`taxonKey` 做连接键，展示时再用 `scientificName` 或 `species`。

## 数据质量提醒

- `species` 可能为空；这通常表示记录只匹配到更高分类阶元，样本中对应 `TAXON_MATCH_HIGHERRANK`。
- `issue` 是分号分隔的质量标记；正式分析时建议显式过滤或记录过滤规则。
- `individualCount` 有缺失，且不等于 checklist effort；将记录数解释为“观测记录数”，不要直接解释为真实种群数量。
- 坐标可能经过四舍五入或解释处理；涉及精细空间分析时要检查 `COORDINATE_ROUNDED`、`COUNTRY_COORDINATE_MISMATCH` 等标记。
- 许可字段样本为 `CC_BY_4_0`，使用或发布结果时要保留来源归属。
